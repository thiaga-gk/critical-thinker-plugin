# Six-Gate Critical-Thinking Workflow

The user supplied a six-level framework that progresses from unreflective thinking to disciplined, continuously improving thought. Operationally, use neutral work-gate names so the workflow describes **what the analysis must do**, not the intelligence of the thinker.

| Operational gate | Source-framework mapping |
|---|---|
| 1. Baseline | Level 1 — Unreflective thinker |
| 2. Challenge | Level 2 — Challenged thinker |
| 3. Structure | Level 3 — Beginning thinker |
| 4. Test | Level 4 — Practicing thinker |
| 5. Integrate | Level 5 — Advanced thinker |
| 6. Decide & Learn | Level 6 — Master thinker |

The gates are cumulative and apply in LIGHT, STANDARD, and DEEP modes. Depth changes; the discipline does not.
For traceability, source Level 6 is the framework's Master thinker level and maps here to **Decide & Learn**.

## Gate 1 — Baseline

**Purpose:** make the default reaction visible before it drives the analysis.

Capture:

- the first answer, preferred solution, or intuitive diagnosis already on the table
- why it feels attractive
- what it presumes to be true
- what problem it appears to solve

**Gate:** the initial answer is documented as a hypothesis, not adopted as the conclusion.

**Common failure:** silently anchoring on the user's proposed solution.

## Gate 2 — Challenge

**Purpose:** expose defects in the current frame.

Test for:

- unsupported assumptions
- ambiguous problem definitions
- selection or confirmation bias
- missing stakeholders or time horizons
- contradictions between claims and evidence
- symptoms being mistaken for causes
- solution language embedded in the problem statement

**Gate:** material thinking defects and decision-critical unknowns are explicit.

**Common failure:** criticizing the idea superficially while preserving the original frame.

## Gate 3 — Structure

**Purpose:** replace ad hoc thought with an explicit method.

Construct:

- a neutral problem statement
- success criteria and non-negotiable constraints
- decision-relevant boundaries
- a small causal hypothesis tree or option set
- required evidence and discriminating questions
- domain-appropriate analysis methods

Possible tools include 5 Whys, cause trees, process maps, Pareto analysis, decision matrices, Bayesian updating, sensitivity analysis, or experiments. Use only tools that materially improve the decision.

**Gate:** a repeatable analysis plan exists and names what evidence could change the view.

**Common failure:** using a named tool as ceremony rather than as a discriminating method.

## Gate 4 — Test

**Purpose:** verify and revise rather than merely structure.

Do the work:

- verify input data, definitions, timestamps, and provenance
- test causal hypotheses against observed patterns
- seek disconfirming cases
- compare predictions with observations
- revise the frame when evidence conflicts
- separate uncertainty from disagreement
- **trace your own quantitative content**: whenever the analysis contains a number, formula, threshold, algorithm, or worked example, execute it by hand and confirm it reproduces and points the right direction — do not accept your *own* calculations at face value

**Quantitative self-verification (mandatory when the analysis has quantitative content):** re-derive every load-bearing number, and run any worked example through the analysis's *own stated method or pseudocode* to confirm the method actually produces the stated result. An example the method cannot reproduce, or a figure that does not reconcile, is a Test-gate failure — repair before proceeding. State in the report that this reproduction was done.

**Gate:** at least one evidence-check/revision loop is shown (or the next discriminating test is specified), **and** any quantitative content has been reproduced against its own method.

**Common failure:** treating the first structured analysis as final; asserting a worked example without ever running it through the algorithm that supposedly generated it.

## Gate 5 — Integrate

**Purpose:** combine systems context, priorities, alternatives, and consequences.

Analyze only the decision-relevant parts of:

- upstream drivers and downstream effects
- stakeholder incentives and perspectives
- interfaces, bottlenecks, and feedback loops
- the critical few issues with greatest decision impact
- alternative explanations and viable options
- trade-offs across cost, time, risk, quality, reversibility, and equity when relevant
- second-order and long-term consequences

**Gate:** the diagnosis or option ranking remains coherent after systems expansion, prioritization, and consequence analysis.

**Common failure:** expanding into a laundry list that does not change priorities or action.

## Gate 6 — Decide & Learn

**Purpose:** turn disciplined thinking into a calibrated, adaptive decision policy.

Produce:

- the recommendation or best current diagnosis
- concise evidence-based rationale
- confidence and uncertainty calibration
- explicit assumptions
- reversible versus irreversible components
- leading indicators and failure signals
- revisit triggers
- learning/feedback loop
- a reasoning summary others can challenge or reuse

**Gate:** the recommendation is actionable, auditable, proportionate, and designed to learn from reality.

**Common failure:** presenting certainty instead of a decision policy that can adapt.

## Mode compression rule

LIGHT may represent all six gates in one compact table. STANDARD and DEEP should use dedicated subsections when the report contract calls for them.

A concise gate is acceptable. A missing gate is not.
